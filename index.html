Domain 1 — 24% of the exam

Cloud Concepts

The "why" of the cloud: value proposition, design principles, migration, and economics.

01

Cloud Value & Benefits

Almost a quarter of the exam. If you only study one domain deeply, study this one.

If you see this on the exam → think this "grows and shrinks" → elasticity "handles more load" → scalability "no upfront cost" → OpEx / pay-as-you-go "launch in minutes, low risk" → agility "bulk discount passed on" → economies of scale
Power grid analogy: On-premises = running your own generator (buy fuel, fix it, guess capacity). The cloud = plugging into the grid (pay for exactly what you use, provider maintains everything).

CapEx → OpEx

On-prem = capital expenditure (huge upfront cost). Cloud = operational expenditure (pay-as-you-go). This shift is the single biggest exam theme in this domain.

The six benefits of AWS Cloud

  • Agility — launch in minutes, kill with no loss
  • Elasticity — auto scale up AND down
  • High availability — redundancy everywhere
  • Global reach — deploy to any region in clicks
  • Economies of scale — AWS's bulk discount passed to you
  • Pay-as-you-go — no contracts, no idle waste
⚠ Exam Trap — Elasticity vs. Scalability

Scalability = adding more lanes to a highway (handling more load over time). Elasticity = lanes that open AND close automatically with real-time traffic (scales up and back down).

Lock it in: Elasticity always implies shrinking back down to save money — that's what separates it from plain scalability on the test.

Cloud Adoption Framework & Migration

How organizations plan a move to AWS, and the strategies/tools used to actually migrate.

Trigger keywords "copy as-is" → Rehost "tinker slightly" → Replatform "redesign cloud-native" → Refactor "no longer needed" → Retire "keep on-prem for now" → Retain "large offline data transfer" → AWS Snowball

AWS Cloud Adoption Framework (CAF)

Groups an organization's readiness into six perspectives, split into Business capabilities (Business, People, Governance) and Technical capabilities (Platform, Security, Operations). Benefits: reduced business risk, improved ESG performance, increased revenue, increased operational efficiency.

RetainKeep it on-premises for now (e.g., not ready, compliance reasons)

Both move bulk data into AWS, and both are correct-sounding.

Lock it in: Question stresses limited bandwidth, a remote site, or petabytes that would take months over the wire → Snow Family (offline, shipped). A usable network connection exists → DataSync (online).
04

Cloud Economics

The financial concepts behind "why moving to the cloud saves money."

Trigger keywords "pay only for usage" → variable cost "already own the license" → BYOL "match instance to need" → rightsizing "bulk hardware discount" → economies of scale

Fixed vs. variable costs

On-prem = mostly fixed costs (buy servers regardless of use). Cloud = mostly variable costs that scale with actual usage.

Licensing: BYOL vs. included

Bring Your Own License (BYOL) = use a license you already own on AWS. Included license = AWS bundles the software license into the hourly rate.

Rightsizing

Matching instance types/sizes to actual workload needs — avoiding paying for unused capacity.

Benefits of automation

An explicit exam-guide bullet. Automation (IaC, auto scaling, scheduled start/stop) cuts manual labor cost, removes human error, and shuts off idle resources you'd otherwise pay for.

Total Cost of Ownership (TCO)

On-prem cost is more than servers: data-center space, power, cooling, network gear, and the staff to run it. The cloud removes those undifferentiated costs.

Economies of scale

AWS buys hardware at massive volume discounts no single company could get alone, and passes savings on to customers.

Security & Compliance

The single heaviest domain. Shared responsibility, IAM, encryption, and the security service lineup.

05

The AWS Shared Responsibility Model

Its own dedicated task statement — expect multiple questions built directly around this model.

Trigger keywords "security OF the cloud" → AWS "security IN the cloud" → you "guest OS on EC2" → customer patches "guest OS on RDS/Lambda" → AWS patches "your data, your IAM" → always the customer
Apartment building: AWS is the landlord — responsible for the building's structure, wiring, plumbing (the physical facility, hardware, network, virtualization layer). You're the tenant — responsible for what happens inside your unit (your data, your locks, who you let in).
AWS is responsible for ("security OF the cloud")You are responsible for ("security IN the cloud")
Physical data centers, hardware, global network infrastructureYour data, classification, and encryption choices
The host operating system & virtualization layerGuest OS, application configuration, and patching (varies by service)
Facility security, power, coolingIAM users, groups, roles, and permissions
Managed service internals (e.g., RDS's underlying OS)Network & firewall configuration (security groups, NACLs)
⭐
The responsibility line shifts by serviceEC2 (IaaS): you manage the guest OS and above. RDS (managed/PaaS-like): AWS manages the OS, you manage data and access. Lambda (serverless): AWS manages almost everything except your code and its permissions.
⚠ Exam Trap — "Who is responsible for X?"

These questions test whether you know the line moves depending on the service.

Lock it in: Data encryption, IAM configuration, and guest-OS patching on EC2 are always the customer's job — no matter how "managed" a service feels, AWS never touches your data or access policies for you.

Deployment Models

Public, private, and hybrid cloud.

Trigger keywords "shared infrastructure" → public "own hardware, full control" → private "connects on-prem to AWS" → hybrid "multiple public clouds, no on-prem" → multi-cloud "dedicated physical line" → Direct Connect
Housing analogy: Public cloud = renting an apartment. Private cloud = owning a house. Hybrid = owning a house + renting a storage unit for the overflow.
ModelKeywordsCost model
PublicElasticity, scalability, no upfront cost, shared infrastructureOpEx
PrivateMaximum control & security, on-premises, big initial investmentCapEx

07

IAM & Access Management

AWS Identity and Access Management sits at the heart of Domain 2. Covers core IAM plus the surrounding identity/secrets ecosystem.

Trigger keywords "permanent credentials" → IAM user "bucket of users" → IAM group "temporary, auto-expiring" → IAM role "SSO across accounts" → IAM Identity Center "store/rotate DB passwords" → Secrets Manager "minimum permissions needed" → least privilege "second factor at login" → MFA "close the account / change support plan" → root user only "sign-up sign-in for my app's users" → Cognito "share resources across accounts" → AWS RAM

User

An identity for a person or app. Long-term credentials (permanent password/keys).

Group

A bucket of users. Apply permissions once to the group instead of one-by-one.

Role

Assumed identity with temporary, auto-expiring credentials. AWS's gold standard for secure access.

Office building security: User = permanent employee ID badge. Group = "marketing access level" applied to a whole department. Role = a visitor pass that only works today and expires at 5pm.
Related serviceWhat it does
IAM Identity Center (AWS SSO)Centralized workforce sign-in across multiple AWS accounts and business apps
AWS Secrets ManagerSecurely stores & auto-rotates secrets like database passwords and API keys
AWS Systems Manager (Parameter Store)Stores configuration data and secrets; also used for operational management/patching at scale
Amazon CognitoSign-up/sign-in and identity for your application's end users (not your staff) — web & mobile apps
AWS Directory ServiceManaged Microsoft Active Directory in AWS; connects existing on-prem AD to AWS
AWS RAM (Resource Access Manager)Share resources (subnets, license configs) across AWS accounts without duplicating them
AWS STS (Security Token Service)Issues the temporary credentials behind every assumed role
Federated identityLetting users sign in with an existing corporate or third-party identity instead of creating an IAM user
Cross-account IAM rolesLet a user or service in one AWS account assume a role in another account — the right way to grant access between accounts instead of duplicating IAM users

Protecting the root user

The exam guide calls this out by name. The root user is the email address the account was created with, and it can do everything — it cannot be restricted by IAM policies.

Lock it down

Enable MFA on root, delete any root access keys, and set a strong password.

Then stop using it

Create an IAM user/Identity Center user for daily work. Never share root credentials.

Tasks only root can do

Close the account, change the account name/email, change or cancel the support plan, restore an IAM user's deleted permissions, register as a seller in Marketplace.

Credential / controlWhat to know
Access keysLong-term programmatic credentials (ID + secret). Rotate regularly; never embed in code or commit to git.
Password policyAccount-wide rules: minimum length, character types, expiration, reuse prevention.
MFASomething you know + something you have. Virtual, hardware, or FIDO security key.
Identity-based policyAttached to a user, group, or role — "what can this identity do?"
Resource-based policyAttached to the resource itself (e.g., an S3 bucket policy) — "who can touch this thing?"
AWS managed vs. customer managedAWS managed = prebuilt and maintained by AWS. Customer managed = you write it, for tighter least privilege.
⚠ Exam Trap — EC2 needs access to S3

Wrong answer: create an IAM user and paste its permanent access keys into the application code.

Lock it in: Correct answer is always to attach an IAM role to the EC2 instance.
⚠ Exam Trap — Cognito vs. IAM Identity Center

Both handle sign-in, so the question hinges on who is signing in.

Lock it in: Your employees accessing AWS accounts → IAM Identity Center. Your customers signing into your app → Cognito.

Security Services

Shield, WAF, GuardDuty, Inspector, and Firewall Manager — two teams: outside guards vs. inside detectives.

Trigger keywords "DDoS / traffic flood" → AWS Shield "SQL injection / XSS" → AWS WAF "ML threat detection, unusual behavior" → GuardDuty "vulnerability scan" → Inspector "central policy, many accounts" → Firewall Manager "find PII / sensitive data in S3" → Macie "investigate root cause of a finding" → Detective
Mansion party: Shield = barricades at the edge stopping a mob. WAF = bouncer checking every guest's ID and bag at the door. GuardDuty = undercover guards watching for suspicious behavior. Inspector = the safety inspector who checks the whole house for weak spots before the party starts.
ServiceRoleTrigger keywords
AWS ShieldDDoS protection (free "Standard" tier for everyone)DDoS, traffic flood, volume attack
AWS WAFWeb application firewall, layer 7 filteringSQL injection, XSS, malicious request content
Amazon GuardDutyContinuous threat detection inside your accountintelligent threat detection, ML, unusual behavior
Amazon InspectorProactive vulnerability scanningvulnerability assessment, scan EC2 for weaknesses
AWS Firewall ManagerCentrally manages WAF rules & security groups across many accountscentral management, multiple accounts, organization-wide policy

The detection pipeline in orderInspector finds weaknesses before attack → GuardDuty detects an active threat → Detective investigates why it happened → Security Hub shows it all in one dashboard. Macie runs alongside, watching your S3 data specifically.

Where to find AWS security information: AWS Security Center, the AWS Security Blog, AWS Knowledge Center, and AWS Artifact for compliance reports. Third-party security products (firewalls, antivirus, SIEM) are available through AWS Marketplace.

⚠ Exam Trap — Shield vs. WAF

"Your app is being hit with cross-site scripting attacks — which service?"

Lock it in: Shield cares about quantity of traffic. WAF cares about quality/content. XSS → WAF.
⚠ Exam Trap — GuardDuty vs. Inspector vs. Macie

All three "find security problems," but each looks at a different target.

Lock it in: Inspector scans workloads (EC2, containers, Lambda) for vulnerabilities. GuardDuty watches account activity/logs for threats. Macie scans S3 data for sensitive content.
09

Encryption: At Rest vs. In Transit

Only two concepts to master here.

Trigger keywords "stationary, stored" → at rest → KMS "moving over a network" → in transit → SSL/TLS via ACM "while uploading" → still in transit
Safe vs. armored truck: Encryption at rest = data locked in a stationary safe. Encryption in transit = data moving inside an armored truck.
StateAnalogyKey AWS service
At restThe safeAWS KMS (Key Management Service)
In transitThe armored truckSSL/TLS via AWS Certificate Manager (ACM)
Key serviceWhat it does
AWS KMSCreate and manage encryption keys; integrates with most AWS services. Keys live on shared AWS-managed hardware.
AWS CloudHSMDedicated, single-tenant hardware security module. You alone control the keys — for strict regulatory requirements.
AWS Certificate Manager (ACM)Provision, manage, and auto-renew free SSL/TLS certificates for encryption in transit.
⭐
KMS vs. CloudHSMDefault answer is KMS. Only pick CloudHSM when the question stresses dedicated/single-tenant hardware, FIPS 140-3 Level 3, or "AWS must have no access to the keys."

S3 encryption at rest options worth recognizing: SSE-S3 (S3-managed keys), SSE-KMS (KMS keys, adds audit trail), and client-side encryption (you encrypt before upload).

⚠ Exam Trap — "Protect data while it's being uploaded to S3"

The word "uploaded" is the whole question — data is still traveling.

Lock it in: Moving = in transit. Only once saved in the bucket does it become at rest.

Global Infrastructure

Regions, Availability Zones, and edge locations.

Trigger keywords "isolated data center" → AZ "group of AZs" → Region "cache near users" → edge location "minimum of 3" → AZs per Region "data must stay in-country" → data sovereignty

Availability Zone (AZ)

One or more physically separate, isolated data centers with their own power, cooling, and networking.

Region

A physical geographic area made up of multiple AZs. AWS's current official minimum is at least three AZs per region.

Edge Location

A smaller site used to cache content closer to end users — powers services like Amazon CloudFront for lower latency.

City of kitchens: A region is a city. Each AZ is a separate pizza kitchen. One oven breaks? The other kitchens keep delivering.
Problem scopeSolution
High availability / fault tolerance (one data center fails)Multiple AZs
Disaster recovery (a whole city/region is hit)Multiple Regions
Data sovereignty (data must stay in a country)Choose specific Region
Latency (serve users fast)Choose nearest Region / use edge locations

Edge & extended infrastructure

The exam guide explicitly asks for "benefits of edge locations" — these are the services that live there or push AWS beyond a Region.

ServiceWhat it is
Amazon CloudFrontAWS's CDN — caches content at 400+ edge locations for low-latency global delivery. Also blocks attacks at the edge alongside Shield/WAF.
AWS Global AcceleratorRoutes users over the AWS backbone via static anycast IPs. Improves performance for non-cacheable, TCP/UDP traffic.
AWS OutpostsPhysical racks of AWS hardware installed in your own data center — true hybrid, for low latency or data residency.
AWS Local ZonesExtends a Region closer to a large metro area for single-digit-millisecond latency.
AWS WavelengthAWS infrastructure embedded inside 5G networks for mobile edge applications.
Regional edge cachesLarger caches sitting between edge locations and your origin, improving cache hit rates.

How to choose a Region

Compliance & data sovereignty

Legal requirements about where data physically lives. This one overrides all others when mentioned.

Latency to users

Pick the Region closest to your customer base.

Pricing

The same service costs different amounts in different Regions.

Service availability

Not every AWS service exists in every Region.

⭐
The magic numberAWS's official minimum is 3 AZs per region.
⚠ Exam Trap — CloudFront vs. Global Accelerator

Both "make things faster globally," so the question hinges on the content type.

Lock it in: Cacheable content (images, video, static sites) → CloudFront. Dynamic, non-cacheable traffic or gaming/IoT/VoIP needing static IPs → Global Accelerator.

High Availability, Deep Dive

The idea underneath nearly every architecture question on the exam.

Trigger keywords "no single point of failure" → high availability "redundancy" → multiple AZs "one AZ, many instances" → still NOT highly available
Solo barista: One employee running a whole coffee shop = single point of failure. Hire a second barista and the shop keeps running if one is out.
⚠ Exam Trap — "10 servers in one AZ, is this highly available?"

No. Even with heavy redundancy, if every instance sits in a single AZ, the whole AZ is still one point of failure.

Lock it in: True high availability = spread across multiple independent AZs.

Compute Services

Containers: ECS vs. EKS

ECS (Elastic Container Service) = AWS's own native container orchestrator. EKS (Elastic Kubernetes Service) = managed Kubernetes. Both can run on EC2 you manage, or serverlessly on Fargate.

Serverless: Lambda & Fargate

Lambda = run code without provisioning any servers. Fargate = serverless compute specifically for containers.

Load Balancers (ELB)

Distributes incoming traffic across multiple targets/instances — improves fault tolerance and availability.

Amazon ECR

Elastic Container Registry — the managed place to store Docker container images that ECS/EKS then run.

EC2 Auto Scaling

Automatically adds and removes instances to match demand. This is what makes a workload elastic.

EC2 Image Builder / AMIs

An AMI is the template (OS + config) an instance launches from.

The rule of one vs. manyOne server + one hard drive = EBS. Many servers sharing one file system concurrently = EFS.

EBS & EFS, in full

Amazon Elastic Block Store (EBS) = block storage volumes for a single instance. Amazon Elastic File System (EFS) = shared file storage many instances mount at once.

Instance Store

Temporary block storage physically attached to the host. Very fast, but ephemeral — data is lost when the instance stops or terminates.

Amazon FSx

Fully managed file systems for specific workloads — FSx for Windows File Server, FSx for Lustre.

AWS Storage Gateway

Hybrid storage — connects on-premises applications to AWS cloud storage.

Lifecycle Policies & AWS Backup

S3 Lifecycle policies automatically transition objects to cheaper tiers as they age. AWS Backup centralizes and automates backups across many AWS services.

AWS Elastic Disaster Recovery

Continuously replicates your servers into AWS so you can fail over quickly during an outage — minimizes downtime and data loss.

AWS Transfer Family

Managed SFTP/FTPS/FTP file transfers directly into S3 and EFS.

⚠ Exam Trap — "Fleet of EC2 instances across multiple AZs needs to access the same data concurrently"

Seeing "EC2 instances" makes people jump to EBS.

Lock it in: EBS only attaches to one instance. "Fleet" + "concurrently" = shared access across many → EFS.
⚠ Exam Trap — Instance Store vs. EBS persistence

A question mentions data must survive an instance stop/restart.

Lock it in: Instance store data is wiped on stop/terminate. If persistence matters, the answer is EBS.
18

VPC & Networking

The bedrock of AWS networking, plus DNS.

Trigger keywords "remembers connections" → stateful → Security Group "checks both directions" → stateless → NACL "domain name → IP" → Route 53
Your own office building in a giant city: The VPC is your own private, logically isolated building. Subnets are the floors. The internet gateway is the front lobby door.

Security Group

Stateful. A personal bouncer at one instance's door. Remembers you on the way in, so it auto-allows the matching traffic back out.

Network ACL

Stateless. A guard for the whole subnet with zero memory. Checks ID coming in AND going out.

Amazon Route 53

AWS's DNS service — translates domain names into IP addresses, registers domains, supports routing policies and health checks. Not a CDN (that's CloudFront).

VPC components

The exam guide asks you to "identify the components of a VPC (for example, subnets, gateways)."

ComponentWhat it does
Internet Gateway (IGW)Attaches to the VPC and allows two-way internet traffic. A subnet is public only if it routes to one.
NAT GatewayLets instances in a private subnet reach the internet outbound only (for patches/updates) while blocking inbound connections. Lives in a public subnet.
VPC Endpoint / AWS PrivateLinkConnects your VPC privately to AWS services without traversing the internet. Keeps traffic on the AWS network.
AWS Transit GatewayA central hub connecting many VPCs and on-premises networks — replaces a tangle of one-to-one peering connections.
VPC PeeringDirect one-to-one private connection between two VPCs.
Amazon API GatewayCreate, publish, and secure APIs at any scale — the common front door for Lambda-backed applications.

Connecting on-premises to AWS

AWS Site-to-Site VPN

Encrypted tunnel over the public internet. Fast to set up, cheap, but subject to internet variability.

AWS Client VPN

Lets individual users/laptops securely connect to your AWS network from anywhere.

AWS Direct Connect

Dedicated private physical line to AWS. Consistent low latency and high bandwidth — but takes weeks to provision and costs more.

⭐
VPN vs. Direct ConnectNeed it today, or cost matters most → VPN. Need consistent, dedicated, high-throughput performance and the question mentions large data volumes or reliability → Direct Connect.
⚠ Exam Trap — Stateful vs. Stateless

Sound similar, behave completely differently on return traffic.

Lock it in: Security Group = stateful = remembers. Network ACL = stateless = forgets, needs rules both directions.
⚠ Exam Trap — "Private instances need to download OS patches"

People reach for an internet gateway, which would expose the instances to inbound traffic.

Lock it in: Outbound-only internet access for a private subnet → NAT Gateway. IGW would make it public.
⚠ Exam Trap — Security Group rules

A question asks how to block a specific malicious IP address using a security group.

Lock it in: Security groups only have allow rules — they cannot deny. To explicitly block an IP, you need a network ACL.

Public vs. Private Subnets

Pulled directly from the official exam guide's list of required topics.

Trigger keywords "route to internet gateway" → public "no route out" → private "path exists ≠ unlocked" → security group still applies
Rooms in your VPC house: Public subnet = the living room, with a front door to the outside. Private subnet = the vault — no direct door out.
⭐
What makes a subnet publicNot its name — only whether its route table routes 0.0.0.0/0 to an internet gateway.
⚠ Exam Trap — Access vs. security

A door existing doesn't mean the door is unlocked.

Lock it in: The route table defines the path. The security group is the guard. You need both.
20

Route Tables

The piece that connects subnets, gateways, and traffic flow.

Trigger keywords "where traffic goes" → route table "outbound rule allows but still no internet" → check the route table
GPS for your VPC city: Destination = the street address you're heading to. Target = the next on-ramp the GPS sends you to.
  • • Every subnet must be associated with a route table.
  • • Unassigned subnets default to the VPC's main route table.
  • • One route table can serve many subnets — but a subnet only uses one route table at a time.
⚠ Exam Trap — "Instance can't reach the internet, but its security group allows all outbound"

The question deliberately rules out the firewall.

Lock it in: Check the route table for a missing internet gateway route.

⚠ Exam Trap — Multi-AZ vs. read replicas

⚠ Exam Trap — Redshift vs. RDS

Lock it in: Day-to-day transactions (OLTP) → RDS/Aurora. Analytics, reporting, data warehouse over huge history (OLAP) → Redshift.

Pricing Models

Golden rule: you only pay for what you use — across compute, storage, and data transfer.

Trigger keywords "flexible $/hr commitment" → Savings Plans "specific instance commitment" → Reserved "reclaimed with warning" → Spot "physical server just for you" → Dedicated Host "data leaving to internet" → costs money

Reserved Instance details

The exam guide calls out RI flexibility and RI behavior in AWS Organizations by name.

ConceptWhat to know

The AWS Free Tier

Always Free

Never expires — e.g., 1M Lambda requests/month, 25 GB DynamoDB storage.

12 Months Free

From account sign-up — e.g., 750 hrs/month of t2.micro EC2, 5 GB S3 Standard.

Trials

Short-term, starts when you activate the specific service.

⚠ Exam Trap — Data transfer direction & scope

Looks too simple — that's how it catches people.

Lock it in: Data transfer in is typically free. Data transfer out costs money. Same-Region transfer is generally cheapest; cross-Region costs more.
⚠ Exam Trap — Reserved Instances vs. Savings Plans flexibility

Both offer similar top-end discounts.

Lock it in: RIs commit to a specific instance/region. Savings Plans commit to $/hr spend and flex across families and regions.
